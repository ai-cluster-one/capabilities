# Doctrine

**Doctrine is constitution, not procedure.** What lives here is principle — the governing forces that fix *what must hold and why*, at the altitude of law. Mechanics never live here: not where a file sits, not the shape of a command, not the name of a helper, not the order of steps. Those resolve to the executable standard ([SHEBANG.md](SHEBANG.md)), the structure ([TEMPLATE.md](TEMPLATE.md)), or the procedure ([ROUTINES.md](ROUTINES.md)); a rule names its principle and points to that home. A statement that reads as a step rather than a law is mis-filed — lift the law out and leave the step in its operational home.

The single source of truth for how capabilities — and the routines that consume them — are built. Every principle and rule lives here, **once**. Each rule states what must hold and why, and carries its own **Validate** clause: the test for whether a given capability meets it. Every Validate clause names how it is enforced — `audit: <check>` (a check `capabilities audit` runs, named so it binds to a live test), `doctor` (the manager's machine reconcile), or `review: <who>` (an agent or a named skill judges it). Nothing restates these elsewhere — [TEMPLATE.md](TEMPLATE.md) applies them to a capability's structure, [ROUTINES.md](ROUTINES.md) applies them to the procedures that consume capabilities, and `capabilities audit` is the mechanical walk of the `audit:` clauses; the judgment calls stay with an agent reading this file. `capabilities selfcheck` keeps the binding honest — a clause tagged `audit` whose check the manager does not perform, or an audit check no clause claims, is itself a violation. Meeting every rule is the bar; an unmet rule is a violation to list.

## The standing rules

1. **One fact, one home.** A fact stated in two places is wrong the moment one copy changes; every fact — a value, a principle, a path — has exactly one home, and everything else references it.
   *Validate (audit: contract-drift; review: agent; review: sanitize-project):* find any fact (an id, a principle, a command, a path) stated in more than one file; each restatement is a violation, consolidated to the single home the others point to. The `sanitize-project` reviewer walks the envelope-side aspect (duplication across a capability's `capabilities/<name>/` files); the agent reviewer covers the capability-side surface.

2. **Just enough at each altitude.** Global is introduction only — one generated skill whose description carries the installed capability names; a capability's own awareness is its `stub` verb's single line, injected per project only where the capability is enabled; detail lives in the on-demand surfaces (`help`, `guide`, the references), never the always-loaded context. Neutrality is tiered with altitude — the declaration (the stub line, the manifest) carries no consumer- or domain-specifics; those belong in the project envelope (connections, identifiers, references).
   *Validate (audit: stub; review: agent):* check the stub line for project specifics or **anything the tool itself answers** (credentials, readiness, the command surface) — it is one plain awareness sentence ending in "run `<name> help`"; check the generated context rule for re-taught tool detail (it composes stub lines, envelope menus, and discovery pointers, nothing more); and check the declaration for domain/consumer specifics that belong in the project envelope; flag bloat or bleed at the wrong altitude.

3. **Discoverable knowledge belongs in the tool, not the docs.** If a fact *should* be answerable by running `<name> help`, it lives in the CLI's own help output, not transcribed into a markdown that goes stale — including the shape of any structured output the tool emits (a contract's fields and envelope), printed by the CLI (e.g. `<name> contract`) and derived from the producer's own builders so it cannot drift. A consumer doc names the home ("run `<name> contract`"), it never copies the shape. The same principle governs a capability's **readiness** — whether it is wired up in the *current* context: that is answered by running `<name> doctor` at use-time, self-reported by the tool, **never inferred from the stub's presence** (the stub is awareness, not a promise the tool is usable here). A `doctor` that fails names what is missing and where to provide it — the global `~/.config/<name>/` or a project-side file — so the remediation, too, is discovered from the tool, not asserted in a doc.
   *Validate (audit: help; review: agent):* find command references or output-shape/schema transcribed into markdown, or a doc that asserts a capability is configured/ready in a context rather than deferring to `<name> doctor`; each belongs in the CLI's help, `contract`, or `doctor`, replaced by a one-line pointer.

4. **Identifiers are non-secret, structural, pure lookup.** Secrets live in env, resolved by the cascade; non-secret connection values (URLs, workspaces, hostnames-with-tenant) live in the connections registry or resolve through the cascade (rule 18); the identifiers asset holds only non-secret structural values — ids, labels, classifications, and breadcrumbs for values not yet pinned — never a secret, and never the narrative of *how* a value is used in a transaction (that treatment is the reference's, rule 11).
   *Validate (audit: ids; review: agent; review: sanitize-project):* flag any secret or connection value sitting in a committed file, and any treatment prose in an identifiers file. The `sanitize-project` reviewer walks the envelope-side identifiers/reference split (both directions) in the consuming project; the agent reviewer covers the capability-side surface.

5. **Link platform docs, don't copy them.** For deep usage of the underlying service, point to its official docs (fetchable live) rather than transcribing; capture only what's specific to *this* capability or *this* project.
   *Validate (review: agent):* find transcribed third-party documentation; replace with a link, keeping only the capability/project-specific part.

6. **Placeholders, never secrets.** Project files install with clearly-marked placeholders; credential files install as `*.example` with empty values; a real secret never enters the repo or a committed file.
   *Validate (review: agent):* scan committed files for real secrets or filled credential values; any is a violation.

7. **Addressed by role, declared once.** A file that needs an asset names it by `(capability, slot)` — "the `<name>` identifiers", "the `<name>` reference" — and resolves it through the capability's own surface (`<name> ids list`, `<name> refs`); it never hard-links the literal path. Across capability boundaries the same holds one level up: a capability needing a *sibling capability's* value names that **capability by name** ("resolved from the `<other>` capability") and resolves into its identifiers, never restating the value or hard-linking the sibling's file. (A capability's own sibling assets may link each other directly — they're declared together.)
   *Validate (review: agent; review: sanitize-project):* find literal asset paths hard-linked by consumers (should resolve through the capability's `ids`/`refs` surface), and any sibling capability's value restated or hard-linked instead of named-and-resolved. The `sanitize-project` reviewer walks the envelope-side aspect (hard links between `capabilities/<name>/` files, sibling-value restatements); the agent reviewer covers the capability-side surface.

8. **Pointers flow down; an asset never names its consumer.** A reference or identifiers file describes the capability for whoever uses it — it never names the routine, workflow, or caller that invokes it. Consumers point *down* to the assets they need; assets never point *up*. A bidirectional link is two homes for one relationship and rots from the upper end.
   *Validate (review: agent; review: sanitize-project):* find any asset that names its consumer (a routine, workflow, or caller), or any bidirectional link; the asset should state what it is and let consumers find it by role. The `sanitize-project` reviewer walks the envelope-side aspect (an envelope file naming its upstream consumer); the agent reviewer covers the capability-side surface.

9. **Domain ownership — which capability a fact belongs to.** A fact lives with the capability whose *domain* it describes, not whichever capability it is *about*. A read-only peer-system capability owns *how the system is read and what its output represents* — the source-side facts. How a **consumer** acts on that output — maps it to the consumer's own accounts, primitives, or treatment — belongs to the **consumer's** capability, even though the fact names the source system. Reading and handling are different domains. This is the horizontal companion to rule 1 (*one* home) and rule 2 (*which* altitude): rule 9 says *which capability*.
   *Validate (review: sanitize-project):* in a capability's docs, find facts from a consumer's domain (the consumer's accounts, primitives, treatment); each is a violation, moved to the consumer's capability — the source capability keeps only how it is read and what its output represents.

10. **A capability never bakes in the consumer's identity.** A capability is portable, so it must not hardcode anything that pins it to one installation — the consuming entity's own name, its organization / tenant / account identifiers, or any value unique to a single consumer. Those live in the consumer's own identity files and resolve at use; the capability refers to the consumer by role ("the consuming entity", "the host project").
    *Validate (review: sanitize-project):* find consumer-specific identity hardcoded in a capability doc; replace with a role reference resolved from the consumer's own files.

11. **Model and procedure each have one home.** The static model — a mapping, a treatment, a taxonomy — lives in the reference (slot 5). The *procedure* that applies it lives in a routine (in the consuming project), or — where it is merely the tool's surface — the CLI's own `help`. The procedure names the model by role and does not restate it; a model copied into both rots from whichever end changes first.
    *Validate (audit: refs; review: agent; review: sanitize-project):* find a model (a mapping, treatment, taxonomy) restated in both a reference and the procedure — a routine — that applies it; consolidate into the reference, the procedure naming it by role. The `sanitize-project` reviewer walks the envelope-side aspect (a step-by-step recipe sitting inside an envelope reference); the agent reviewer covers the capability-side surface.

12. **Affirmative, self-contained framing.** A doc states what it *is*, in the present — not what it is not, what it *was*, or what a neighbouring system does ("never through X", "unlike Y", "formerly Z"). Minimise cross-referencing; keep a pointer only for a strong, specific routing reason. Change history lives in git, not the prose; a sibling system documents itself.
    *Validate (review: agent; review: sanitize-project):* find prose that defines by negation, by history ("formerly", "moved from"), or by a neighbour's behaviour; restate it affirmatively. The `sanitize-project` reviewer walks the envelope-side aspect (this framing appearing inside a `capabilities/<name>/` file); the agent reviewer covers the capability-side surface.

13. **Host-neutral declaration, manager-owned composition, one host injector.** A capability *declares* itself once, host-neutrally, through its own contract verbs — `stub`, `manifest --json` — and the manager snapshots that declaration into the registry (`~/.capabilities/<name>/`) at install. The capabilities manager alone decides how the system introduces itself and renders the enabled declarations, registry snapshots, and project envelopes: `capabilities context --fragment` is the host-neutral Markdown provider surface. Host-specific placement is owned by exactly one selected project context manager, never by an individual capability. In a standalone project the capabilities manager also fills that role: `capabilities context --claude` places the block at `.claude/rules/CAPABILITIES.md`, while `capabilities init --codex` wires the generated Codex artifacts and `capabilities context --codex` refreshes them. In a project bound by `.contextkit/config.toml`, ContextKit owns the combined host files and hooks, invokes `capabilities context --fragment`, and inserts its output verbatim; it does not reimplement the gate, snapshot, or envelope renderer. Capabilities init and the host-writing context modes therefore create no competing hooks or standalone context there and hand refreshes to `contextkit build`. Global awareness remains the manager-generated `capabilities` skill (`~/.claude/skills/capabilities/` and `~/.agents/skills/capabilities/`). No project-root instruction markdown file is required or assumed. Every generated file is written by its owning manager, **never authored or edited by the agent**. The CLI surfaces by a symlink onto `PATH`. Another host context manager swaps only placement while the manager-owned capability block, registry, and CLI-on-`PATH` route stay the same. This is the host-portability companion to rule 10 (consumer-portability): rule 10 keeps a capability free of one consumer's identity; rule 13 keeps it free of one host's wiring.
    *Validate (audit: manifest; review: agent):* find a capability that bakes host placement into itself; find a host context owner that reads the capability gate, snapshots, or envelopes and composes its own capability block instead of inserting `capabilities context --fragment`; or find capabilities-owned host injection beside ContextKit-owned injection in a ContextKit-bound project. Each is a violation, consolidated to the manager-owned provider block and the selected host owner's single placement mechanism.

14. **A capability's guides have one upstream home, fetched live.** How to author a specific kind of content *with this tool* is consumer-neutral and capability-scoped: it belongs to the capability, not to any one consuming project, and its single home is the file in the capability's **upstream repository**. The CLI is its surface (`<name> guide [topic]`): the topic list is computed from what the capability ships so it cannot drift, and a body is fetched **live** so an upstream edit reaches every consumer at once. The upstream file is the authority; a local cache is its offline floor. The resolution mechanics — the conditional request, where the cache lives, the fallback order — are defined once in the executable standard ([SHEBANG.md](SHEBANG.md#guides)), as the credential cascade is. This is the within-capability companion to rule 5: rule 5 fetches the *underlying service's* docs live; rule 14 does the same for a capability's *own* guidance.
    *Validate (audit: guide; review: agent):* find capability authoring-guidance copied into a consuming project's assets or transcribed into an always-loaded doc, or a `guide` command that serves a bundled file as authoritative instead of resolving live against the declared upstream with cache-fallback; each is a violation, consolidated to the single upstream home and surfaced by the CLI.

15. **One writer per shared file.** Every machine-managed file has exactly one writer. The **manager** is the sole writer of the registry and its snapshots, the project gate (`capabilities/settings.json`), `capabilities/.gitignore`, migration from the legacy `.capabilities/` tree, the generated skill, and the hook wiring. A **capability script** is the sole writer of its own state home and its identifiers envelope. `credentials.env` is **human-written only** — a script never writes back into it; what a login mints lives in the state home (rule 16). A file with two writers rots from whichever end wrote last.
    *Validate (review: agent):* find a capability script writing a manager-owned file, anything other than a human writing `credentials.env`, or any file with two writers; each is a violation, consolidated to the declared writer.

16. **State follows the scope of the credentials that minted it.** A capability declares its credential scope in the manifest, and what those credentials mint — session cookies, scrape caches, pending login state — lands in the matching state home: project scope → `capabilities/<name>/state/`, user scope → `$XDG_STATE_HOME/<name>/`. Project state is never committed; the manager-owned `capabilities/.gitignore` guards it and `capabilities doctor` verifies the guard. A session cookie is a secret, so the guard is doctrine-level, not housekeeping. The resolution code is the executable standard's ([SHEBANG.md](SHEBANG.md#state)).
    *Validate (doctor; review: agent):* find minted state written into a committed file or into `credentials.env`, state in a home that contradicts the declared scope, or a project `capabilities/` directory without the gitignore guard; each is a violation.

17. **The gate is the project's word on a capability.** Whether a capability is visible and runnable in a project is read from the manager-owned gate file, in three states: **absent** — runs, not injected (ad-hoc use keeps working: home dir, servers, cron); **enabled** — runs and injected; **disabled** — not injected, and every verb exits 4 with an envelope that tells the agent to ask the user, never to enable it itself. Visibility requires explicit enable; the hard block fires only on explicit disable. The gate is a guardrail, not a security boundary; its mechanics are the executable standard's ([SHEBANG.md](SHEBANG.md#the-project-gate)).
    *Validate (audit: gate; review: agent):* confirm every verb of a disabled capability — `help` and `doctor` included — exits 4 with the ask-the-user envelope, that an enabled-but-broken capability degrades to a single line in the injection rather than failing the build, and that nothing but the manager writes the gate file.

18. **Configuration resolves as connections; a gate is not a value.** Every capability resolves its configuration as named connections. With no registry there is one implicit connection, `default`, resolved by the bare cascade; a registry — the project envelope's `connections.json`, else the user config home's, first found authoritative, never merged — defines the set: a `default` pointer plus entries whose non-secret values sit literally and whose secrets are named by env-key indirection, still resolved by the cascade. Selection is deterministic — explicit flag, else the default pointer, else the sole entry, else a refusal as ambiguous — and `<name> connections` reports every connection's resolution (tier, source, secrets masked) without touching the network. A connection's writability resolves from two declarations — its entry's `allow_write`, else the capability's `WRITE_DEFAULT`, declared `false` where a write leaves the system (messages a human, sends mail, moves money) — and a connection resolving to read-only is a source: its write verbs exit 4, and nothing in the cascade — no flag, no env var — lifts the gate, because the cascade resolves values and a gate is policy. Mechanics are the executable standard's ([SHEBANG.md](SHEBANG.md#connections)).
    *Validate (audit: connections, connections/core-only; review: agent):* confirm selection follows the deterministic order and refuses ambiguity rather than guessing; that no secret sits in a registry entry (env-key indirection only); that a write verb on a read-only connection exits 4 with the ask-the-user envelope and no flag or env var lifts the gate; that a capability whose writes leave the system declares `WRITE_DEFAULT = False`; and that `<name> connections` reports resolution with secrets masked and no network I/O.

## The credential cascade

Every capability's executable resolves its credentials with the **same deterministic cascade** — a contract the CLI implements in code, not merely documents, because it is how config follows the developer from a laptop (project and user config present) to a deployed host (global env only) with no change in behaviour and no model lookup. The tiers, their order, the reason process env sits below the user file, and the shared code that realizes it are defined once in [SHEBANG.md](SHEBANG.md#the-credential-cascade) — the executable's authoring standard. The principle the doctrine fixes: resolution is deterministic and identity-free (rule 10), and secrets resolve from env, never from a committed file (rule 4).

Each capability **declares its credential scope** in the manifest, and install scaffolds accordingly: `project` (the default) for service- and tenant-bound tools — the secret lands in the consuming project's `.env(.local)`, nothing global; `user` for personal-account tools where one human has one identity — `~/.config/<name>/credentials.env`. The tiers and their order are the same under either scope; the scope says where the scaffold puts the key. Inline credentials in dialogue ride tier 4 — an env-prefix invocation — and a secret never appears on argv: no `--token`-style flags exist.

*Validate (audit: manifest, connections, doctor; review: agent):* confirm the executable implements the SHEBANG.md cascade in order (flags → project `.env` → user config → process env, first non-empty wins), that the identifiers asset points to env for secrets rather than holding them (rule 4), and that the manifest declares the capability's **credential scope** (`project` / `user`) matching where its required config actually resolves — and that `doctor`, not a doc, is what reports readiness in a given context (rule 3), judging it through the same cascade resolution the work verbs use rather than a parallel check.

## Deviations are allowed — and recorded

These rules are a strong default, not a cage. A capability may deviate **with justification**. When it does, the deviation + reason is recorded in the capability's **own dedicated deviation file** — a file whose sole purpose is to hold it, kept apart so it is never commingled with other content or accidentally removed — so a later audit reads it as a **deliberate choice or a known edge case**, not drift to "fix." The validator advises; it does not force one path. A recorded, justified deviation is not a violation.
