# emta — deviations

This file's sole purpose is to hold `emta`'s deliberate, justified departures from the [SHEBANG](../../SHEBANG.md) defaults, kept apart so an audit reads them as choices, not drift (DOCTRINE — *[Deviations are allowed — and recorded](../../DOCTRINE.md#deviations-are-allowed--and-recorded)*). Each realizes the *intent* of the pattern in the terms of a service that has **no public API and no durable credential** — `maasikas.emta.ee` (the Estonian Tax Board customer portal) is reached as a logged-in browser: a SmartID + Mobile-ID ceremony mints a short-lived PAS bearer inside the browser session; every authed call to this capability sends that bearer directly.

## Empty `CRED_KEYS` with `STATE=True` and a live remote

The cascade has nothing to resolve. SHEBANG's credential pattern presumes a durable env-resolvable secret (token / username+password) that `doctor`'s preflight gate can declare absent; this capability has none. The session is minted by a SmartID + Mobile-ID ceremony in the browser and reaches the script as a paste — a cURL copied out of devtools — never as an environment variable. Persisting that cURL into the cascade would be a longer-lived, lower-visibility secret in plain `.env`/credentials files; refusing it is the privacy choice. The intent — credentials are declared, scoped, and the cascade picks them up — is realized in this transport's terms: the declared store is the per-connection state file under `.capabilities/emta/state/<conn>/session.json` (mode `0600`, gitignored by the manager-owned guard), populated by `emta session ingest` reading stdin. `doctor`'s credential preflight is structurally vacuous (zero keys, zero missing); readiness collapses entirely onto the round-trip — `GET /payments-and-settlements-api/client/session-user` with the stored bearer + xsrf + cookie jar — which is what proves the session is alive anyway.

## Paste-fed bearer persisted in state and used directly — no programmatic refresh

The PAS bearer is **the** credential. It is paste-fed via `emta session ingest` (extracted from the cURL's `Authorization: Bearer …` header, or equivalently the `PAS-AUTHORIZATION` cookie — Chrome puts the same value in both) and persisted alongside the cookie jar and XSRF token in the per-connection state file (mode `0600`). Every authed call sends the stored bearer directly — `Authorization: Bearer …` header plus `PAS-AUTHORIZATION` cookie — together with `PAS-XSRF-TOKEN` (as cookie + `x-xsrf-token` header) and the rest of the PAS-path cookie jar (`customer-kmd2_JSESSIONID`, `cf_clearance`, `_cfuvid`). There is no `/customer-portal/access-token` mint loop: a PAS-path cURL does not carry `JSESSIONID` (that cookie is path-scoped to `/customer-portal/`), so the mint endpoint cannot be reached from the data the script has. There is no programmatic refresh — any `401` or `403`/`406` from any authed call is reported as `session_stale` (exit `2`) with one consistent remediation hint: run `emta session ingest` and paste a fresh `Copy as cURL` from any logged-in `/payments-and-settlements-api/` XHR. The intent of SHEBANG's auth-altitude exit-code rule — auth is always distinguishable so a caller knows to re-authenticate — holds at this transport's grain: there is exactly one auth altitude (the browser session), one stored credential (the bearer + its cookie/xsrf scaffold), and one remediation (paste a fresh cURL). Stored bearer is treated as a secret on the same discipline as the cookies: never logged, never printed, never echoed, never committed.

### `principal_person_id` is auto-discovered via `/client/session-user`

The portal's `/payments-and-settlements-api/client/session-user` endpoint is the bootstrap and the probe: it is the only authed call that needs no `x-principal-person-id` header, and on `200` its body carries `data.principal.principalId` — exactly the value every data call needs in that header. `emta session ingest` and `emta doctor` both call it: ingest stores the bearer + xsrf + cookies, then bootstraps via session-user and writes the discovered `principalId` to the ids store (as a string); doctor refreshes the same value on every healthy probe. The human is never asked for `principal_person_id` and never has to paste a cURL whose `x-principal-person-id` header carries it. (`emta ids set principal_person_id <value>` remains available as a manual override; the resolver coerces numeric ids values to string so a value stored as a JSON number works the same as one stored as a JSON string.)

## Stdin-only secret ingestion

`emta session ingest` reads the cURL command from stdin and **refuses argv**: a TTY-attached invocation prints the usage and exits 6. The raw cURL line carries the live bearer, the XSRF token, and the PAS cookie jar — handing it on argv leaks it into the parent shell's history, the host's process list, and any tooling that snapshots them; stdin contains it to the pipe. The intent of SHEBANG's secrets discipline (DOCTRINE rule 16) — secrets land only in the declared store and never on routes that persist them elsewhere — is realized at the one ingestion seam the capability owns.
