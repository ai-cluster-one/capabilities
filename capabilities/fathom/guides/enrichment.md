# Enrichment — generating meeting summaries with `fathom`

The `fathom` CLI is a pure adapter: it syncs, stores, reads, and searches
meetings. It does not interpret content. **Summaries are generated by the host
agent**, not by the tool — the agent reads a stored transcript, writes a
summary, and saves it back through `fathom update`. This guide is the
procedure for doing that well; the orchestration itself belongs to a routine
in the consuming project (cadence, which meetings, when), which names this
procedure by role.

## The contract the tool gives you

- `fathom list --has-transcript true` — meetings that have a transcript.
  Each carries `has_summary`; the ones with `has_summary: false` need one.
- `fathom read <id>` — the stored meeting as JSON. The `transcript` field is
  the text to summarise; `language` is the language to write the summary in.
- `fathom update <id> --summary "…"` — saves the summary and stamps
  `enriched_at`.

## One meeting, one subagent, strictly sequential

Generating a summary fills context with a full transcript. To protect the
main conversation, **delegate each summary to its own subagent**, and run them
**one at a time** — never in parallel, never queued ahead. Each meeting is
fully summarised and saved before the next one starts.

```
for each meeting needing a summary:
    1. Spin up ONE subagent for that recording id
    2. Subagent: `fathom read <id>` → reads the transcript JSON
    3. Subagent: generates the summary from the transcript
    4. Subagent: `fathom update <id> --summary "…"` → saves it
    5. Subagent completes and is destroyed (its context, and the
       transcript, are freed)
    6. Main thread confirms success, then proceeds to the next meeting
```

Do not run multiple summary subagents concurrently or in the background
together, and do not queue the next before the previous has saved.

## Subagent prompt template

```
You are processing a single meeting transcript. Follow these steps exactly:

STEP 1 — read the transcript:
    fathom read <RECORDING_ID>
  The output is JSON; the `transcript` field holds the text, `language`
  the language code.

STEP 2 — generate the summary. It must be:
  - Plain text: no markdown, no headings, no bullet points.
  - 3-5 sentences capturing who participated, what was discussed, and what
    was decided or agreed as next steps.
  - Written in the SAME LANGUAGE as the transcript (read the `language`
    field — "ru" → Russian, "et" → Estonian, "en" → English, …).
  - Factual and specific — name concrete topics, not vague descriptions.
  - No timestamps, no speaker attributions.

STEP 3 — save it:
    fathom update <RECORDING_ID> --summary "The summary text here…"

STEP 4 — report: the recording id, the number of speakers, and the summary.
```

## Where the summary then lives

Once saved, the summary is searchable: `fathom scan <query>` matches summary
text and returns a snippet around the hit, and `fathom search <query>` matches
it alongside titles and participants. A meeting's summary is regenerated by
running `fathom update <id> --summary "…"` again — it overwrites and re-stamps
`enriched_at`.
