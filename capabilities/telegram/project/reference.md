# Telegram — reference

> **Purpose.** The standing home for *project-specific* operational context: prose about how **this project** uses Telegram, where that's neither a value nor the command surface. What each chat means here, an export or transcription convention, a retention or naming policy, which chats map to which outputs.
>
> Structural values (chat ids, usernames) live in `identifiers.md`; how the CLI and Telegram behave — the command surface, the credential cascade, the stateful startup protocol, the chat-reference forms, the export JSON shape, the I/O envelope + exit codes — lives in `telegram help` (project-agnostic, never copied here).
>
> An *executable* read/export/transcribe flow (which chats sync where, on what trigger) is a **project routine** in `.routines/`, not this file; point to it from here, don't embed it.
>
> Ships empty on purpose, so the home is always labeled and no agent has to decide whether it should exist or what goes in it. Populate it only when real context accrues — an empty reference is conformant, not a gap. Replace this note with that content when it arrives.
